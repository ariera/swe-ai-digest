Simon Willison

Co-creator of the Django web framework and independent developer whose blog is arguably the highest-volume, most rigorous public writing on LLMs from a practitioner's perspective.

GLM-5.1: Towards Long-Horizon Tasks (2026-04-07)

Simon Willison evaluates Z.ai's GLM-5.1, a 754B-parameter MIT-licensed open-weight model. In his standard 'pelican on a bicycle' SVG benchmark, the model unpromptedly generated an HTML page with embedded CSS animations — a novel behaviour for an open-weights model. When the animation broke, the model correctly diagnosed the CSS/SVG transform conflict and fixed it autonomously. Willison judges this the best SVG output he has seen from an open-weights model. For engineers tracking open-weight frontier models, GLM-5.1 represents a notable step in multi-step reasoning, self-correction, and creative code generation within a single prompt/follow-up exchange, and its MIT licence and OpenRouter availability lower the barrier to experimentation.

Anthropic's Project Glasswing - restricting Claude Mythos to security researchers - sounds necessary to me (2026-04-07)

Willison covers Anthropic's Project Glasswing, under which their unreleased Claude Mythos model is being made available only to vetted security partners. Mythos has already found thousands of high-severity vulnerabilities across major OSes and browsers, autonomously chained multi-step exploits, and wrote working browser exploits at a near-zero failure rate compared to prior models. Anthropic and partners including AWS, Apple, Microsoft, and Google are pooling $100M in usage credits and $4M in donations to get ahead of the inevitable proliferation of these capabilities. For software engineers, this marks a qualitative threshold: AI-assisted vulnerability discovery is no longer theoretical, and the security posture of long-lived C codebases is under systematic automated pressure.

Google AI Edge Gallery (2026-04-06)

Willison reviews Google's AI Edge Gallery, the first official app from a major model vendor for running models (Gemma 4 E2B and E4B) entirely on-device on an iPhone. The E2B model, at 2.54GB, is described as fast and genuinely useful. The app supports image Q&A, audio transcription, and a tool-calling 'skills' demo against eight HTML-based interactive widgets. Willison notes the absence of persistent conversation logs as a significant UX gap. For engineers interested in on-device AI deployment, this represents a meaningful milestone: a production-quality, official vendor app that demonstrates multimodal tool-use inference running locally on consumer hardware without a server.

Eight years of wanting, three months of building with AI (2026-04-05)

Willison highlights Lalit Maganti's essay on building syntaqlite — a production-grade SQLite linting/parsing library — over three months using Claude Code, after eight years of procrastination. The key findings: AI was effective at overcoming the initial inertia of tedious grammar-rule implementation and generating concrete prototypes quickly, but actively harmful for architectural decisions. Deferred design choices corroded codebase clarity, and AI's inability to evaluate subjective design questions led to wasted weeks on dead ends. The first AI-generated prototype was thrown away; the final library required extensive human-led design work. This is one of the more nuanced practitioner accounts of AI-assisted development, distinguishing where agents excel (implementation, prototyping) from where they fail (architecture, open-ended design).

scan-for-secrets 0.1 (2026-04-05)

Willison describes scan-for-secrets, a Python CLI tool he built using README-driven development and Claude Code with TDD. The tool scans directories for secrets (API keys, etc.) and their common encodings (JSON-escaped, backslash-escaped variants), helping engineers safely publish Claude Code session transcripts. The development workflow — writing a detailed README first, then delegating implementation to a coding agent using red/green TDD — is itself the main AI-relevant insight. Willison demonstrates how a well-specified README doubles as a prompt specification for an agent, enabling rapid, test-verified tool creation. This is a practical example of human-in-the-loop agentic engineering for security-adjacent developer tooling.
